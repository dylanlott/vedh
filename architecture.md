# Architecture

> A breakdown of the vEDH.xyz architecture

## Games and Boardstates

The Go server is responsible for synchronizing games and boardstates.

Boards and games are stored separately because storing them together increases contention and locking concerns. Keeping them separate allows each player to have a single source of truth for their game and makes logic for joining and dropping easier.

The server tracks a map of games and boardstates in memory and persists their storage to the database. This is one area that could be improved with an optimized approach to reduce the memory footprint and rely more heavily on the Postgres storage layer.

## Persistence

The persistence layer is handled by Postgres and uses a model generated by 99designs/gqlgen to interface it with GraphQL queries and mutations. The persistence layer connects to the database and handles all of the CRUD operations for games and boardstates.
